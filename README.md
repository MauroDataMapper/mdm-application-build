# Mauro Data Mapper Application

This is the repository to use to build the MDM application.
Currently it only builds the backend, the front end would needed to be added manually.

## Requirements

* Grails 5.1.1+
* Gradle 7.3.3
* Java 17

All of the above can be installed an easily maintained by using [SDKMAN!](https://sdkman.io).

*Grails 4.0.6 will not run in any version above Java 12, and Grails 4 will not work on any version below Java 11*

## Building

**Please note this whole build system is still a work in progress and may not start up as expected,
also some properties may not be set as expected**

Currently you will need to

1. Build the backend using `grails war`
1. Extract the war file to `build/application`
1. Build the mdm-ui using `ng build --prod`
1. Copy the contents of the `dist` folder to `build/application`
1. Repack `build/application` as a war file or drop the folder into a Tomcat instance.

### Adding plugins to the build

If you want to add any plugins from the [Mauro Data Mapper Plugins](https://github.com/MauroDataMapper-Plugins) organisation then you simply need to add them as Gradle dependencies to the `dependencies.gradle` file in the root of this repository.
They will then be added to the build when you call `grails war`.

### Changing application properties

Any properties which need to be added or altered from the defaults can be added to the `grails-app/conf/application.yml` file.
Calling `grails war` will then package these properties into the new war file.

## URL Endpoints

In the provided mappings report file there are several generic properties which can have various possibilities depending on what you're trying to obtain.

The application has the following concepts, this is a rough definition:
* container : This contains models, it is securable resource.
* catalogueItems : These are all items which can be catalogued
  * models : These are securable and define the type of catalogue item catalogued, they all belong to a container; nominally a folder.
    * modelItems : These define items inside a model, they all belong to a model and are non-securable resources.
* facets : These are non-securable resources, they all belong to a container, model or model item.

When looking to replace the generic properties it is preferred that you use the full endpoint type; e.g `dataModels` not `dataModel`.

#### Examples

* `/api/${catalogueItemDomainType}/${catalogueItemId}/annotations`
  * ` /api/dataModels/${catalogueItemId}/annotations`
  * ` /api/dataClasses/${catalogueItemId}/annotations`
  * ` /api/primitiveTypes/${catalogueItemId}/annotations`
* `/api/${catalogueItemDomainType}/${catalogueItemId}/metadata`
  * ` /api/dataModels/${catalogueItemId}/metadata`
  * ` /api/dataClasses/${catalogueItemId}/metadata`
  * ` /api/primitiveTypes/${catalogueItemId}/metadata`
* `/api/${securableResourceDomainType}/${securableResourceId}/securableResourceGroupRoles`
  * `/api/folders/${securableResourceId}/securableResourceGroupRoles`
  * `/api/dataModels/${securableResourceId}/securableResourceGroupRoles`
* `/api/tree/${containerDomainType}`
  * `/api/tree/folders`
